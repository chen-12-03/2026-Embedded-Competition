# GitHub Issue 管理流程（团队执行版）

本流程用于团队在 GitHub 上进行任务分派、进度同步与阶段验收。

## 1. 管理方式

- 主系统：GitHub Issues + GitHub Projects（Board）
- 原则：每个任务一个 Issue，按看板状态流转，不用额外日报表

## 2. 项目看板建议

在仓库创建 Project（Board 视图），建议列：

1. Backlog
2. This Week
3. In Progress
4. Blocked
5. Done

## 3. Issue 创建规范

每个 Issue 至少包含：

- 目标：本任务要达成什么
- 验收标准：完成判定条件（可验证）
- 截止时间：计划完成日期
- 负责人：单一 owner
- 风险/依赖：阻塞点、外部依赖

标题建议格式：

`[模块] 动词 + 结果`

示例：

- `[采集] 完成温湿度传感器驱动接入`
- `[网关] 实现本地告警页面阈值配置`

## 4. 日常执行节奏

每日每人最小动作（1 分钟）：

1. 将卡片拖到正确状态列
2. 在 Issue 留一句当日进展
3. 若阻塞，写明原因和所需支持

## 5. 半周/每周验收

按半周进行一次小验收，按周进行一次集中验收。

验收只认可追溯证据：

- 代码链接（PR/commit）
- 文档链接
- 测试结果或运行截图
- 日志/数据文件

无证据不标记 Done。

## 6. 常见问题约束

- 不要把多个大任务塞到同一个 Issue
- 不要仅写“已完成”而没有验收证据
- Blocked 必须写阻塞原因与解除条件
- Done 任务必须可回溯到具体交付物
